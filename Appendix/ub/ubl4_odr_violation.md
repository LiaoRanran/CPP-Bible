# UB-L4 ODR violation（单定义规则违反）

**分类**：链接/语义 UB ｜ **检测工具**：链接器 / LTO / 优化器（常静默）
**源码**：[ub_odr_violation.cpp](./ub_odr_violation.cpp)

---

## ① 问题代码（跨翻译单元的真正 ODR 违反）

```cpp
// ---- h.hpp（被 a.cpp 与 b.cpp 包含）----
inline int config() { return 1; }     // a.cpp 的翻译单元看到此定义
// 但 b.cpp 包含的是另一个版本：
inline int config() { return 2; }     // ❌ 同一 inline 实体两处定义不一致 → ODR 违反
```

跨 TU 编译：
```
g++ -std=c++23 -O2 -c a.cpp -o a.o
g++ -std=c++23 -O2 -c b.cpp -o b.o
g++ a.o b.o main.o -o prog      # 链接器通常不报错（inline 弱符号各取其一）
./prog                         # 行为未定义：可能每次调用取到不同定义，或优化器假定一致而错
```

## ② 编译器行为（本机实测，单 TU 版）

单 TU 编译本库 `ub_odr_violation.cpp`：**无错误，正常运行**（单文件无法复现跨 TU 违反）。
ODR 违反的本质是**跨翻译单元**，单文件不构成。

## ③ 为什么危险

C++ 要求：同一实体（函数/变量/类/模板实例）在整个程序中有**唯一定义**（inline /
模板允许"多份相同定义"，但必须**逐词一致**）。一旦不一致，链接器因弱符号机制
常常**静默选一份**，但优化器可假定"所有定义等价"并做跨 TU 内联——结果不可预测：
同一调用点有时返回 1、有时返回 2，或崩溃。

## ④ 常见触发点

| 触发 | 说明 |
|------|------|
| inline 函数多份不一致 | 如上例 |
| 头文件宏改变实体定义 | 不同 TU 用不同宏展开出不同定义 |
| 模板特化/变量模板不一致 | 同特化两处定义不同 |
| 匿名命名空间放进头文件 | 每 TU 各得一份"不同"实体（非 ODR 违反，但是重复定义来源）|

## ⑤ 修复

```cpp
// 1) 保证 inline / 头文件实体在任何 TU 逐词一致（同一份头文件）
// 2) 跨 TU 共享状态用 `inline` 变量（C++17）或 `extern` 声明 + 单 TU 定义
// 3) 开启 LTO（-flto）让优化器跨 TU 看见冲突，更易暴露
// 4) 头文件用 include guard / #pragma once 防重复包含导致意外多定义
```
