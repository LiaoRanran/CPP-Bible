# 第 1 章 C 与 C with Classes：C++ 的基因

> 本章定位**：历史与哲学**。理解 C++ 为何是"带类的 C"、为何坚持零开销、
> 为何保留与 C 的兼容，是后续一切语言特性的认知地基。
> 模板说明：本章为历史章，省略"汇编分析""内存模型（逐特性）"等元素（无单点特性可析），
> 但在 1.4 给出 C with Classes 对象布局的等价内存图。

**本章依赖**：无（全书起点）。
**后续相关**：第 3 章（各标准版本）、第 22 章（类与对象基础）、第 39 章（虚函数/对象模型）。

---

## ① 定义

**C with Classes** 是 Bjarne Stroustrup 于 1979 年在 Bell 实验室为增强 C 语言而设计的
前身语言，引入了**类、派生类（继承）、强类型检查、内联、默认参数、构造函数**等机制。
它是 C++ 的直接祖先（1983 年更名 C++）。其核心承诺：**在不牺牲 C 的运行效率与贴近硬件
能力的前提下，提供更好的抽象与组织大规模代码的能力**。

---

## ② 历史背景：C 的诞生与成功

- **1969–1972**：Ken Thompson 与 Dennis Ritchie 在 Bell 实验室开发 Unix；Ritchie 将
  B 语言（源于 BCPL）改造为 C，以便用高级语言重写 Unix 内核。
- **1972**：C 语言基本定型；它长在 PDP-11 上，直接映射硬件：指针=地址，数组=连续内存，
  结构体=内存块。
- **1978**：《The C Programming Language》（K&R）出版，成为事实标准，推动 C 跨平台传播。
- **成功要素**（决定 C++ 必须兼容 C 的根本原因）：
  1. **贴近硬件**：能写操作系统、驱动、嵌入式。
  2. **高效**：编译为紧凑机器码，无运行时包袱。
  3. **可移植**：相对汇编，C 源码跨机器可重编译。
  4. **生态**：Unix 与大量系统软件用 C 写成，代码资产巨大。

> [经验] C++ 从第一天起就不打算"取代 C"或"另起炉灶"，而是要**吸收 C 的全部资产**
> 并向上叠加抽象。这是它区别于 Pascal/Modula/Ada 等"理想化"语言的关键战略。

---

## ③ 为什么出现：C 的局限

当软件规模从"几千行"膨胀到"数十万行"（如网络模拟、分布式系统、GUI），纯 C 暴露痛点：

| C 的局限 | 具体表现 | 工业后果 |
|---------|---------|---------|
| 无原生封装 | 数据与方法分离，靠 `struct` + 函数约定，易破坏不变式 | 大型项目中不变式频繁被绕过 |
| 无继承/复用 | 相似数据结构需复制代码或宏技巧 | 代码膨胀、维护灾难 |
| 弱类型检查 | 函数声明可缺省（`implicit int`）、指针随意转换 | 链接期才暴露错误，调试痛苦 |
| 无构造/析构 | 资源（内存/句柄）手动管理，易泄漏 | 长运行服务稳定性差 |
| 无运算符/泛型 | 容器（如向量）需 `void*` + 宏，类型不安全 | 泛型数据结构脆弱 |

Stroustrup 在写他的博士课题（分布式系统仿真，用了 Simula 的类机制）时，深切体会到
**Simula 的抽象能力**与 **C 的效率**无法兼得。他要在 C 上"嫁接" Simula 的类。

> [经验] 设计动机不是学术优雅，而是**工程刚需**：写一个能跑、能扩展、不慢的仿真系统。

## ④ C with Classes 引入的关键机制（1979–1983）

按时间线给出机制与对应现代 C++ 的落点：

| 年份 | 机制 | 解决的问题 | 现代等价 |
|------|------|-----------|---------|
| 1979 | `class` + 成员函数 | 封装数据与操作 | `struct`/`class`（第22章） |
| 1979 | 派生类（单继承） | 代码复用/层次建模 | `class D : public B`（第38章） |
| 1980 | 强类型检查 | 编译期抓错 | 类型系统（第13章） |
| 1980 | 构造函数/析构函数 | 资源自动初始化/清理 | RAII（第35章） |
| 1981 | 内联函数 `inline` | 替代宏、零开销抽象 | `inline`（第20章） |
| 1981 | 默认参数 | 简化接口重载 | 默认实参（第20章） |
| 1982 | 引用 `&` | 安全别名、运算符重载需求 | 引用（第14章） |
| 1982 | `const` | 只读契约、接口约束 | `const`（第15章） |
| 1983 | 虚函数 `virtual` | 运行时多态 | 多态（第39章） |
| 1983 | 函数/运算符重载 | 同名不同参的接口统一 | 重载（第20/24章） |
| 1983 | `new` / `delete` | 类型安全的动态分配 | 内存管理（第29章） |
| 1983 | 更名 **C++**（`++` = 自增） | 品牌化 | — |

> [标准] 注意：上述机制在 C with Classes 时代尚无 ISO 标准，属于**实现层**行为。
> 它们后来被 C++98 标准化，细节有演化（如 `new` 的数组 cookie）。

## ⑤ 设计哲学：四条不变量（贯穿全书）

1. **零开销抽象（Zero-overhead principle）**
   > "You don't pay for what you don't use. What you do use, you couldn't hand-code any better."
   > —— Stroustrup
   抽象（类、虚函数、模板）在**不使用时零成本**；在使用时，编译器生成的代码不劣于
   手写最优 C。这是 C++ 区别于 Java/C#（有 GC、有运行时）的根本。

2. **静态类型安全**
   尽量在编译期捕获错误。C 的"宽松"被收口，但保留显式逃逸舱口（`reinterpret_cast`、
   `union`、`void*`）。

3. **与 C 的高度兼容**
   C++ 可以（大部分）直接编译 C 代码、调用 C 库（`extern "C"`）。这让它继承了 C 的生态。

4. **直接贴近硬件**
   不隐藏内存与机器模型。指针仍是地址，你可以决定每一字节。

> [对比其他语言] Rust 也追求零开销+安全，但以"所有权在编译期"换取内存安全；
> Go 用 GC 换简单；Java/C# 用托管运行时换生产率。C++ 把控制权交还程序员。

## ⑥ 对象模型雏形（C with Classes 风格）

C with Classes 的类在内存上等价于一个 `struct` 加上成员函数表（虚函数尚未普遍，早期靠
"函数指针成员"模拟）。一个最简向量类：

```cpp
// industrial: 一个固定容量、类型安全的整数向量（C with Classes 风格，C++23 可编译）
#include <cstddef>
#include <stdexcept>

class IntVector {
    int*   data_   = nullptr;   // 堆上存储
    size_t size_   = 0;
    size_t capacity_ = 0;
public:
    IntVector(size_t cap) : data_{new int[cap]}, capacity_{cap} {}
    ~IntVector() { delete[] data_; }          // 析构：自动释放

    void push_back(int v) {
        if (size_ == capacity_) throw std::length_error("IntVector full");
        data_[size_++] = v;
    }
    int  at(size_t i) const {
        if (i >= size_) throw std::out_of_range("bad index");
        return data_[i];
    }
    size_t size() const { return size_; }
};
```

**等价内存布局图**（对象实例，不含堆上的 `data_` 数组）：

```
  IntVector 实例（栈上，24 字节 x86-64）
  ┌──────────┬──────────┬──────────┐
  │ data_    │ size_    │ capacity_│   int*=8B, size_t=8B
  └────┬─────┴────┬─────┴────┬─────┘
       │          │          │
       ▼          │          │
   [堆] int[cap]  │          │
       0  1  2 ...│          │
                  ▼          ▼
             运行时计数    容量上限
```

对比纯 C 写法（需手动 `malloc/free`、易漏 `free`、类型用 `void*` 不安全），类的价值立现：
**构造即分配、析构即释放，不变式（`size_<=capacity_`）由成员函数维护**。

## ⑦ 工业影响（为什么这套基因赢了）

- **系统软件**：数据库（MySQL/PostgreSQL 部分）、浏览器引擎（Chromium）、操作系统组件
  仍大量 C++，因要贴近硬件又不放下抽象。
- **游戏/图形**：Unreal、Unity(C++)、游戏引擎几乎全 C++，性能敏感且抽象复杂。
- **高频交易**：微秒/纳秒级延迟要求零开销，C++ 是唯一选择。
- **嵌入式**：汽车 ECU、航天、医疗设备，资源受限但需抽象。
- **今日遗产**：你写的每一行 `class`、每一个 `new`、每一次虚函数调用，基因都可追溯至
  1979 年的 C with Classes。

---

## ⑧ 标准化前夜（1983–1998）

- 1983 更名 C++；1985《The C++ Programming Language》第一版；CFront（C++→C 转译器）让
  C++ 能在只有 C 编译器的机器上跑。
- 1980 末–1990 初：多厂商各自实现（Borland Turbo C++、Microsoft、IBM、GCC  nascent），
  方言分化（如模板、`bool` 类型、异常的支持先后不一），可移植性下降。
- 1990：ANSI X3J16 成立；1991 与 ISO WG21 合并，启动标准化。
- 1998：**C++98** 发布，模板、异常、RTTI、STL 入标准，C++ 进入"标准时代"。
  → 详见第 3 章逐版本详解。

## ⑨ 面试题

1. **"C++ 是 C 的超集吗？"** 答：不是。C++ 删改了部分 C 语义（如 `void*` 不能隐式转
   非 `void*` 指针、更严格的隐式声明），且 C99/C11 的某些特性 C++ 不采纳。多数 C 代码
   可编译为 C++，但反向不成立。
2. **"为什么 C++ 不去掉指针、改用 GC？"** 答：违背零开销与贴近硬件原则；存量系统代码
   与实时性要求不允许不可控的 GC 停顿。
3. **"零开销抽象具体指什么？"** 答：见 1.5 第 1 条；常考 `vector::operator[]` 是否比
   C 数组访问慢（不慢，二者同构）。
4. **"C with Classes 解决了 C 的哪些痛点？"** 答：封装、继承、构造/析构（RAII 雏形）、
   强类型。

## ⑩ 最佳实践（从历史得到的现代启示）

- **仍用 RAII**：1979 的构造/析构思想，今天仍是管理资源的第一原则（第 35–36 章）。
- **别为兼容而写 C 风格**：默认用 `class`/`std::vector`/`std::unique_ptr`，而非
  `malloc`/`raw array`。
- **理解"为何兼容"**：与 C 库互操作时用 `extern "C"`（第 20 章），但新代码不必模仿 C。

---

## ⑪ 常见错误（初学者常踩）

- 把 C++ 当"带类的 C"，只用类却仍手写 `new/delete`、不写析构 → 泄漏。应直接用智能指针。
- 认为"C++ 比 C 慢" → 同算法同抽象层级下基本同速；慢通常源于误用（如 `std::function`
  装箱、过度虚调用），见第 21/147 章。
- 混淆"零开销"与"自动优化" → 零开销是**抽象无额外税**，不是"编译器会替你写好一切"。

---

## ⑫ 性能分析（简述）

C with Classes 时代已验证：类成员函数（非虚）被内联后与 C 函数调用无差；虚函数引入一次
间接调用（约 1–5ns，见第 39 章数据）。**抽象成本可控且可测量**。

## ⑬ 示例（工业级，非 Hello World）

**示例 1：C 风格 vs C++ 风格的资源管理对比**（体现 1979 动机）

```cpp
// C 风格：易漏 free，类型用 void* 不安全
typedef struct { int* buf; size_t n; } CVec;
CVec* cvec_new(size_t n){ CVec* v=malloc(sizeof(CVec)); v->buf=malloc(n*sizeof(int)); v->n=n; return v; }
// 调用方必须记得 cvec_free，否则泄漏 —— 大型项目中的典型 bug 源

// C++ 风格：构造/析构自动配对（RAII 雏形源自 C with Classes）
struct CppVec {
    std::vector<int> buf;          // 用标准容器，析构自动释放
    explicit CppVec(size_t n): buf(n) {}
};
```

**示例 2：沿用 1980 内联思想的零开销抽象**（避免宏）

```cpp
// 1981 引入 inline 取代函数宏；今天仍如此
inline int clamp(int v, int lo, int hi) { return v < lo ? lo : v > hi ? hi : v; }
// 调用 clamp(x,0,255) 在 -O2 下无函数调用开销，优于 #define CLAMP(v) ...（无类型检查）
```

**示例 3：派生类（1979 继承）建模真实层次**

```cpp
struct Shape { virtual ~Shape() = default; virtual double area() const = 0; };
struct Circle : Shape {
    double r;
    double area() const override { return 3.14159265358979 * r * r; }
};
// 多态源自 C with Classes 的"派生类"；1983 加 virtual 使其运行时可分派
```

## ⑭ 对比其他语言

| 语言 | 抽象机制 | 运行开销模型 | 与 C 兼容 |
|------|---------|------------|----------|
| C | 无（struct+函数） | 零 | 自身 |
| C++ | 类/模板/泛型 | 零开销（可选） | 高 |
| C# | 类/泛型 | GC + JIT | 低（P/Invoke） |
| Java | 类/泛型（类型擦除） | GC + JVM | 低（JNI） |
| Rust | trait/泛型 | 零开销 + 所有权 | 中（FFI） |
| Go | 接口/结构体 | GC | 中（cgo） |
